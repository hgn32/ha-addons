# ha-addons プロジェクトルール

## HAアドオンのリリース手順

アドオンをリリースする（`config.json` のバージョンを上げる）ときは、上流（Suwayomi等）の
新バージョン対応かどうかに関わらず**常に**、以下を必ず行うこと。

1. **`config.json` のバージョンを上げる**
   - `"version"` フィールドをインクリメントする（例: `"0.12"` → `"0.13"`）

2. **`CHANGELOG.md` を更新する**（バージョンを上げたら必ず変更履歴を書くこと）
   - アドオンディレクトリ内の `CHANGELOG.md` に新バージョンのエントリを追記する
   - **保持する変更履歴は最新3バージョン分までとする。** それより古いエントリは
     削除する（HA の UI で更新履歴が長くなりすぎないようにするため。削除した
     古い履歴も git のコミット履歴には残る）
   - 形式（常に最新3件だけを残す）:
     ```
     ## <新バージョン>

     - <変更内容>

     ## <1つ前のバージョン>

     - ...

     ## <2つ前のバージョン>

     - ...
     ```
   - このファイルはHome AssistantのUI上でアドオンの更新履歴として表示される

3. **マージ後、GitHub Actions がエラーなく動作することを確認する**
   - main へマージしたら、GitHub Actions（イメージのビルド・publish 等）が
     エラーなく完了することを必ず確認する
   - 失敗している場合は原因を調べて修正し、成功するまで対応すること

## shell スクリプト（run.sh・cont-init.d・services.d 配下）を変更するときの検証義務

- **「ロジック的に正しいはず」という推測だけでリリースしてはならない。** `set -e` 系の
  起動スクリプトは、途中の1コマンドが失敗するだけでサービス全体が無限クラッシュ
  ループに陥る。実際にこのリポジトリで「チェーンが既に存在する」→「REJECT ター
  ゲットがカーネルモジュール不足で失敗する」と、直したはずの箇所の**次の行**で
  また同じパターンのクラッシュが立て続けに発覚したことがある。モックでのシミュ
  レーションだけで「直った」と判断して出荷した結果、繰り返しユーザーに実機で
  クラッシュを踏ませてしまった。
- 変更前に「Docker は使えない／手元に環境が無い」と決めつけて確認を省略しない。
  このセッションでは `dockerd` を起動し、実際に対象アドオンの Dockerfile から
  イメージをビルドし、コンテナ内で該当スクリプトを実行することが実際に可能
  だった（プロキシ経由のパッケージ取得が必要な場合は `/root/.ccr/README.md` の
  docker build 節を参照し、CA バンドルを組み込んで `--network host` でビルドする）。
  「試していないのにできないと言う」のは禁止。
- 起動スクリプト（`run.sh` / `cont-init.d/*` / `services.d/*/run` など）に変更を
  加えたら、リリース前に最低限以下を行うこと。
  1. `bash -n` で構文チェック
  2. 変更した分岐・失敗パスを、可能な限り**本物の iptables / wg-quick 等の
     コマンドを使って**（モックではなく）意図した通りに動く/失敗時にクラッシュ
     しないことを確認する。ホストに対象コマンドがあればホスト上で、無ければ
     `docker build` して実際にコンテナを起動し、スクリプトを直接実行して確認する
  3. 「1回目の起動」だけでなく、**再起動・クラッシュ後の再実行を想定して2回以上
     連続実行**し、状態が残ることで2回目以降だけ失敗するパターン（チェーンの
     重複作成など）が無いか確認する
  4. `set -e` 環境下で、途中のコマンドが失敗した場合にスクリプト全体がどう終了
     するか（無限ループにならないか、エラーが握りつぶされず HA のログタブに
     残るか）を確認する
- 上記の実機検証をせずに「ロジックは合っているはず」で config.json のバージョンを
  上げてリリースすることを禁止する。

## ユーザーへの案内・機能設計の原則

- ユーザーは基本的に Home Assistant の画面（アドオンのログタブ・設定タブ・
  Ingress パネル等）以外にアクセス手段を持たない前提で考えること。
  `docker exec`、SSH、コンテナ内のファイルを直接開く、といった操作を
  ユーザーに要求する案内や実装を作らない（例:「/data/xxx.log を見てください」
  とだけ言って終わらせない）。
- ログや状態など、ユーザーに確認してほしい情報は、HA のログタブに出力するか、
  アドオン自身の Ingress パネル等、HA の画面内で完結する手段で見られるように
  実装すること。ファイルパスを案内するだけで「見る手段」を実装しないのは禁止。
