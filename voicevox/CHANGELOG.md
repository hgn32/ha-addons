## 0.11

- 【診断】起動監視ログを追加。原因切り分けに必要な情報をすべてログタブに出力します
  - 起動時に `options.json` の内容と実際の起動コマンド(CORS 設定含む)を表示
  - ポート 50021 で待ち受けを開始するまで、15 秒ごとに経過時間・実メモリ(RSS)・仮想メモリ(VSZ)・設定上限を表示
  - 待ち受け開始時に「エンジン起動完了(所要秒数・エンジン版)」を表示。この行が出るまで 50021 は Connection refused になります
  - エンジンプロセスが終了した場合は必ず exit code をログに記録(これまで無言で止まっていたケースを可視化)
- 【修正】`max_memory_mb` の既定値を `0`(無制限)に変更。0 より大きい値を設定した場合のみ `ulimit -v` を適用(上限も 16384 に拡大)
  - 0.10 イメージの実測で、エンジンは起動完了までに仮想メモリ(VSZ)を **約 8.1GB** 要求することを確認(実メモリ RSS は約 2.4GB)。このため 0.10 の既定 2048MB では `memory allocation failed` で abort、8000MB でもわずかに不足して起動不能だった
  - 制限を設定する場合は 12288 以上を推奨(実測ピーク 8044MB + 余裕)
- 【追加】`load_all_models` オプション(既定 `true`=従来どおり)。`false` にすると全モデルの事前読み込みを行わず数秒で待ち受けを開始するため、モデル読み込みフェーズの問題かどうかを 1 操作で切り分けできます
- 【追加】`preload_speaker_ids` オプション。`load_all_models: false` と組み合わせると、指定したスタイル ID のモデルだけを読み込みます(実測: 全モデル読み込み RSS 約2.4GB/VSZ 約8.0GB → スタイル1件のみ RSS 約0.24GB/VSZ 約1.3GB)。未指定のスタイルも初回利用時に自動読み込みされるため使用可能です
